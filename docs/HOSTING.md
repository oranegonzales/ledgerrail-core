# Free hosting plan

LedgerRail Core uses two persistent services for the public demo and an ephemeral Kafka service in CI, so the portfolio can run without a virtual machine or a time-limited managed-broker trial.

| Component | Provider | Free-tier role |
| --- | --- | --- |
| Java API | Render | Builds and runs the Docker container |
| PostgreSQL | Neon | Stores transfers, ledger entries, and outbox events |
| Kafka verification | GitHub Actions + Testcontainers | Starts a disposable broker and verifies delivery on every change |

## Why InfinityFree is not used

InfinityFree supports PHP and MySQL or MariaDB websites. LedgerRail requires a persistent Java process, a Docker build, PostgreSQL, and later Kafka connectivity. InfinityFree can host a separate static portfolio page, but it cannot run this API.

## Render limitations

The Free web-service instance provides 512 MB RAM and 0.1 CPU. It spins down after 15 minutes without inbound traffic and can take about one minute to wake. Its filesystem is temporary.

The application is configured for these limits:

- Maximum Java heap of 256 MB
- Serial garbage collector
- Maximum Hikari connection pool of five
- No application data stored on the local filesystem
- Health check at `/actuator/health`

The public demo should be described as a cloud-deployed portfolio sandbox, not a production payment system.

### External uptime monitoring

An HTTP monitor with an interval shorter than 15 minutes will usually prevent an idle Render service from spinning down, but it does not turn a Free instance into an always-on service. Render can still restart Free instances, and a running service consumes the workspace's 750 monthly Free instance hours.

For this project, monitor the lightweight liveness endpoint instead of the database-aware health endpoint:

```text
https://ledgerrail-core.onrender.com/actuator/health/liveness
```

UptimeRobot Free checks every five minutes. Keeping one service awake for a full 31-day month uses about 744 of the 750 available hours, leaving almost no allowance for another Free web service. Use the monitor during active job applications or demonstrations, and pause it when continuous wakefulness is not useful.

## Create the Neon database

1. Create a Neon Free project named `ledgerrail`.
2. Keep the default root branch for the deployed demo.
3. Open the Connect dialog and disable Connection pooling.
4. Keep the hostname, database, username, and password private.
5. Use an SSL-enabled JDBC URL in this format:

```text
jdbc:postgresql://YOUR-NEON-HOST/YOUR-DATABASE?sslmode=require
```

Flyway creates the schema when the API starts.

The direct connection is intentional because Flyway and the application share one datasource. HikariCP limits the service to five connections, which is appropriate for a single free Render instance. A later milestone can separate Flyway onto a direct connection and use Neon's pooled hostname for application traffic.

## Deploy to Render

1. Push the repository to GitHub.
2. In Render, create a Blueprint from the repository.
3. Select the Free instance type if Render asks for a plan.
4. Enter the following private environment values:

| Name | Value |
| --- | --- |
| `DATABASE_URL` | Neon JDBC URL with `sslmode=require` |
| `DATABASE_USERNAME` | Neon database role |
| `DATABASE_PASSWORD` | Neon role password |
| `PORTFOLIO_API_KEY` | Long random key generated by Render or a password manager |
| `PUBLIC_DEMO_ENABLED` | `true` to allow anonymous synthetic transfer endpoints |
| `PUBLIC_DEMO_REQUESTS_PER_MINUTE` | Optional per-client limit; defaults to `60` |
| `PUBLIC_DEMO_WRITES_PER_DAY` | Optional persistent global limit; defaults to `500` |

5. Deploy and wait for `/actuator/health` to return `UP`.
6. Open the Render-generated URL and confirm the LedgerRail dashboard appears.

Do not place database credentials, the portfolio API key, or future broker credentials in GitHub.

## Add a custom domain later

Render supplies a public `onrender.com` address and managed TLS. A custom domain can be attached later without changing the application. The temporary `forvermine.lat` domain is not required for the first deployment.

## Kafka deployment position

The public Render service does not connect to a managed Kafka provider. The providers evaluated for this portfolio exposed time-limited credits rather than a dependable ongoing free environment, and running Kafka inside Render's 512 MB container would be unreliable.

Version 0.3.0 includes the broker-neutral outbox publisher, producer acknowledgements, bounded retries, stale-claim recovery, terminal failure state, event-ID de-duplication, and reconciliation. `KAFKA_ENABLED=false` and `KAFKA_CONSUMER_ENABLED=false` keep both Kafka components inactive on Render. The CI workflow runs `mvn verify`, starts PostgreSQL and Apache Kafka containers, and proves the complete delivery and reconciliation path. See [`KAFKA.md`](KAFKA.md) for the delivery model and optional configuration contract.
